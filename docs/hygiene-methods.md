# Каталог методов гигиены (AGENTS.md)

Что пользователь чёрного ящика (Claude Code) может контролировать **без изменения
цикла агента**. Это рычаги Run 2 воркшопа — участники применяют их к раздутому
`AGENTS.md` и измеряют дельту относительно Run 1.

> **Связанные документы:** [charter.md](./charter.md) — архитектура воркшопа;
> [mcp-proxy-methods.md](./mcp-proxy-methods.md) — каталог методов прокси (Run 3).

---

## Почему AGENTS.md вообще влияет на токены

AGENTS.md — это **системный промпт**, который Claude Code отправляет **каждый ход**.
Каждый лишний символ в нём умножается на количество ходов. При 10 ходах и
AGENTS.md размером 5K символов (~1,250 токенов):

```
Стоимость AGENTS.md = 1,250 токенов × 10 ходов = 12,500 токенов
Из них: input (если не в кеше) или cache-read (если префикс стабилен)
```

Но проблема не только в объёме — раздутый AGENTS.md **направляет агента не туда**:
- Широкие инструкции → агент исследует весь репозиторий
- Нет ограничений вывода → агент пишет эссе перед каждым действием
- Дорогая модель по умолчанию → каждый токен стоит в 5 раз дороже

## L0 — Model Choice (выбор модели)

### Логика

Per-token price spread между моделями Anthropic — примерно **5×** (Haiku → Opus).
Выбор самой дешёвой модели, которая всё ещё решает задачу корректно — самый
прямой рычаг снижения стоимости.

### Как применить

В `.claude/settings.json`:
```json
{ "model": "claude-sonnet-4-6" }
```

Или в самом Claude Code: `/model`

### Выигрыш

| Модель | Цена input (за 1M) | Цена output (за 1M) | Стоимость Run 1 (~165K токенов) |
|---|---|---|---|
| Opus 4 | $15 | $75 | ~$3–5 |
| Sonnet 4 | $3 | $15 | ~$0.60–1 |
| Haiku 4 | $1 | $5 | ~$0.20–0.35 |

При одинаковом количестве токенов — **↓ 80% стоимости** только за счёт модели.

### Риск

Слишком слабая модель не решает задачу → агент делает больше ходов → экономия
съедается ретраями. **Нужно проверять quality gate после смены модели.**

### Частотность

Применяется **один раз** перед запуском. Эффект на **каждый токен** прогона.

## L1 — Context Scoping (сужение контекста)

### Логика

AGENTS.md с широкими инструкциями («исследуй репозиторий», «найди проблему»)
заставляет агента читать десятки файлов. Каждое чтение добавляет содержимое файла
в контекст → контекст растёт → каждый последующий ход дороже.

Сужение: назвать **конкретные файлы** и **конкретную задачу**.

### Пример

```
❌ Плохо (раздуто):
   "Explore the codebase and fix any bugs you find in the search functionality.
    Look at all related components, services, and tests. Consider edge cases
    like empty states, error handling, race conditions, and performance."

✅ Хорошо (scope):
   "Fix catalog search in apps/angular-demo/src/app/catalog/. The controller
    (catalog-search.controller.ts) has a debounce bug and stale-response race.
    The component (catalog-search.component.ts) has incomplete states (idle,
    loading, empty, error). Do not change catalog-api.ts types or add deps."
```

### Выигрыш

| AGENTS.md | Файлов прочитано | Средний размер контекста к 5-му ходу |
|---|---|---|
| Раздутый (без scope) | 15–25 | ~50K токенов |
| Со scope | 4–6 | ~12K токенов |
| **Сокращение** | ↓ 70% чтений | **↓ 75% контекста** |

### Сценарий

- Задача имеет чёткую область (конкретный модуль, директория, файл)
- Репозиторий большой (>100 файлов) — без scope агент теряется
- **Не применять** когда область задачи действительно широка (рефакторинг across
  modules) — тогда scope вредит

## L2 — Output Discipline (дисциплина вывода)

### Логика

Output токены стоят **в 5 раз дороже** input. Агенты по умолчанию склонны писать
«here's my plan…», «let me think about this…», «I'll now proceed to…» — это чистые
потери. Явное указание «минимальные изменения, без преамбул» сокращает output.

### Пример

```
❌ Плохо:
   "Fix the bugs and explain your reasoning."

✅ Хорошо:
   "Apply minimal changes. No preamble. No explanations unless something fails.
    Output a git diff only."
```

### Выигрыш

| Дисциплина | Output токенов (типичный прогон) |
|---|---|
| Без ограничений | ~3,000–8,000 |
| «Без преамбул, diff only» | ~800–2,000 |
| **Сокращение** | **↓ 60–75% output** |

В денежном выражении при output ×5: экономия значительна даже при скромном
сокращении токенов.

### Сценарий

- Задача чисто механическая (добавить debounce, поправить типы)
- Агент уже показал что понимает задачу (после первого хода)
- **Не применять** когда агент должен объяснить архитектурное решение

## L3 — File Hints (явное указание файлов)

### Логика

Вместо того чтобы агент сам искал файлы (через grep/search), явно указать их в
AGENTS.md. Каждый search-вызов стоит токенов на tool schema + результат поиска.
Прямая ссылка на файл экономит 1–3 хода.

### Пример

```
❌ Плохо:
   "Find and fix the search bug."

✅ Хорошо:
   "Files:
    - apps/angular-demo/src/app/catalog/catalog-search.controller.ts (the bug)
    - apps/angular-demo/src/app/catalog/catalog-search.component.ts (the UI)
    - apps/angular-demo/src/app/catalog/catalog.types.ts (types, don't change)
    - apps/angular-demo/src/app/catalog/catalog-search.controller.spec.ts (tests)"
```

### Выигрыш

| Подход | Поисковых вызовов | Ходов до первого чтения |
|---|---|---|
| Без hints | 2–4 (grep/search) | 3–5 |
| С hints | 0 | 1 |
| **Сокращение** | ↓ 2–4 вызова | **↓ 2–4 хода** |

### Сценарий

- Задача локализована в известных файлах
- Репозиторий имеет похожие имена в разных местах (search может найти не то)
- **Не применять** когда вы не знаете где баг — тогда hints могут ввести в
  заблуждение

## L4 — Cache Discipline (дисциплина кеша)

### Логика

Cache-read стоит **~10% от input**. Стабильный системный промпт (AGENTS.md) + один
и тот же набор файлов в начале контекста → они обслуживаются из кеша, а не как
дорогой input.

Что разрушает кеш:
- Переключение модели между ходами
- Изменение AGENTS.md между ходами
- `/clear` без необходимости
- Переупорядочивание контекста

### Как применить

Не дёргать модель и конфиг без нужды. Если меняете AGENTS.md — делайте это **до**
запуска агента, а не во время сессии.

### Выигрыш

| Режим | Доля cache-read |
|---|---|
| Стабильный префикс, одна модель | 70–90% |
| Частые смены модели, `/clear` | 10–30% |

Разница: при 70% cache-read input по цене ~0.1× — эффективная стоимость контекста
↓ в 3–5 раз.

### Сценарий

- Всегда. Это не разовая оптимизация, а дисциплина в течение всей сессии.

## L5 — One Clear Up-Front Prompt (точный первый запрос)

### Логика

Каждый ход переотправляет историю. Если первый запрос vague («почини поиск») —
агент потратит 3–4 хода на исследование, и каждый ход будет нести багаж предыдущих.
Точный первый запрос с задачей, ограничениями и файлами → решение за 1–2 хода.

### Пример

```
❌ Плохо (vague):
   "Read TASK.md and fix it."

✅ Хорошо (precise):
   "TASK.md describes a bug in catalog-search.controller.ts. Apply these fixes:
    1. Add debounce (300ms) to search input
    2. Normalize whitespace (trim + collapse)
    3. Cancel stale responses (switchMap)
    4. Add idle/loading/empty/error states to the component
    Do not change catalog.types.ts or add runtime dependencies.
    Report what changed when done."
```

### Выигрыш

| Первый запрос | Ходов до решения | Общий токен-расход |
|---|---|---|
| Vague | 5–8 | ~50K |
| Precise | 1–3 | ~12K |
| **Сокращение** | ↓ 60% ходов | **↓ 75% токенов** |

### Сценарий

- Задача чётко определена (acceptance criteria известны)
- Вы уже знаете где баг и как его чинить
- **Не применять** когда вы передаёте агенту исследовательскую задачу без чётких
  критериев

## L6 — Minimal CLAUDE.md (минимальный конфиг агента)

### Логика

CLAUDE.md (в отличие от AGENTS.md) — это **проектные инструкции**, которые
применяются ко всем задачам в репозитории. Раздутый CLAUDE.md наказывает **каждую**
сессию, даже не связанную с текущей задачей.

### Пример

```
❌ Плохо (5000+ chars):
   "# Project Instructions
    This is an Angular design-system app with the following architecture...
    [10 параграфов описания архитектуры]
    [список из 50 компонентов]
    [правила код-ревью на 3 страницы]
    [инструкции по деплою]"

✅ Хорошо (300 chars):
   "# Project Instructions
    Angular app. Commands: npm start, npm run test.
    Task: read TASK.md → pull the ticket from jira MCP.
    Edit only the feature dir named in the ticket."
```

### Выигрыш

| CLAUDE.md размер | Токенов в ход | ×10 ходов | Экономия vs 5000 chars |
|---|---|---|---|
| 5,000 chars | ~1,250 токенов | 12,500 | — |
| 400 chars | ~100 токенов | 1,000 | ↓ 92% |

### Сценарий

- Всегда. CLAUDE.md должен быть минимальным — детали задачи в TASK.md.

## L7 — Subagents (суб-агенты)

### Логика

Claude Code позволяет определять суб-агентов в `.claude/agents/*.md`. Каждый
суб-агент получает **узкий контекст** (только нужные ему файлы и инструкции) и
возвращает родителю **только результат**, а не полный транскрипт.

Это гибридный рычаг: настройка — на уровне конфига (гигиена), но эффект — на
уровне цикла (инженерия). Суб-агент изолирует контекст и не даёт ему «загрязнять»
основной диалог.

### Пример

```
.claude/agents/test-runner.md:
  "You are a test runner. Run: npm test -- --testPathPattern=<suite>.
   Return only: pass/fail count + list of failed test names. No stdout."

.claude/agents/code-reviewer.md:
  "You are a code reviewer. Review the git diff. Return only:
   critical issues (if any) + approval status. No inline suggestions."
```

Использование в AGENTS.md:
```
"After making changes, delegate to test-runner agent to verify.
 Then delegate to code-reviewer agent for final check."
```

### Выигрыш

| Сценарий | Без суб-агента | С суб-агентом | Сокращение |
|---|---|---|---|
| Прогон тестов (100 тестов, 50K stdout) | 50K в основном контексте | ~200 chars (pass/fail count) | ↓ 99.6% контекста |
| Код-ревью (diff 5K) | 5K в основном контексте | ~100 chars (approved/critical) | ↓ 98% контекста |
| Поиск по документации | Результаты в контексте | Только релевантные ссылки | ↓ 85% контекста |

### Сценарий

- Повторяющиеся операции с предсказуемым результатом (тесты, линтер, сборка)
- Операции требующие чтения больших объёмов (поиск по документации, анализ логов)
- **Не применять** когда суб-агенту нужен полный контекст родителя (тогда spawn
  съест больше чем сэкономит)

### Частотность

2–5 вызовов суб-агентов за прогон. Каждый вызов изолирует потенциально большой
контекст.

## L8 — Skills (подключаемые инструкции)

### Логика

Skills (`.claude/skills/*.md`) — это модульные инструкции, которые загружаются
**только когда релевантны** задаче. В отличие от AGENTS.md (загружается всегда),
skills лежат на диске и активируются по триггеру (ключевые слова в задаче, тип
файла, явный запрос).

### Пример

```
.claude/skills/angular-expert.md:
  "You are an Angular (v17+) expert. Key patterns for this repo:
   - Use RxJS switchMap for cancellation, not async/await
   - Components use OnPush change detection
   - Tests use Jest + jest-preset-angular
   - State: BehaviorSubject + Observable, not signals (legacy)
   Load when: task mentions Angular, RxJS, or component."

.claude/skills/debounce-pattern.md:
  "For debounce implementations:
   - Use RxJS debounceTime(300) + distinctUntilChanged
   - Always pair with switchMap to cancel stale requests
   - Never use setTimeout for debounce in Angular context
   Load when: task mentions debounce, search, or input handling."
```

### Выигрыш

| Подход | Инструкций в промпте | Когда загружены |
|---|---|---|
| Всё в AGENTS.md | 7 тем, 3K chars | Всегда (каждый ход) |
| Skills по требованию | 1–2 темы, 0.4K chars | Только когда релевантны |

При 10 ходах: AGENTS.md 3K × 10 = 30K токенов vs skills 0.4K × 10 = 4K → ↓ 87%.

### Сценарий

- Домен-специфичные знания (Angular, React, Go, Kafka, PostgreSQL)
- Повторяющиеся паттерны (debounce, error handling, auth flow)
- **Не применять** для инструкций, нужных в каждой задаче — их место в AGENTS.md

## L9 — Lazy Tool Loading (ленивая загрузка инструментов)

### Логика

По умолчанию все MCP-инструменты загружаются при старте агента и их схемы
отправляются **каждый ход**. Lazy loading позволяет агенту загружать инструменты
только когда они нужны — через отдельные `.mcp.json` файлы для разных фаз задачи:
discovery, fix, verify.

### Пример

```
Вместо одного .mcp.json со всеми 26 инструментами GitHub:

.mcp-discovery.json — только search_repositories, search_code (2 инструмента)
.mcp-fix.json       — create_branch, push_files, create_pull_request (3)
.mcp-verify.json    — list_commits, get_pull_request_status (2)

Агент подключает нужный набор на каждой фазе.
```

### Выигрыш

| Подход | Инструментов в промпте | Схем в ход |
|---|---|---|
| Все сразу | 26 | ~3,500 токенов |
| По фазам | 2–3 | ~300–500 токенов |
| **Сокращение** | ↓ 90% инструментов | **↓ 85% схем в ход** |

### Сценарий

- Много MCP-серверов (3+ серверов, 15+ инструментов)
- Чёткое разделение задачи на фазы (discovery → fix → verify)
- **Не применять** когда агент переключается между фазами часто — переподключение
  стоит дороже экономии

## L10 — Hooks (событийная автоматизация)

### Логика

Claude Code поддерживает хуки (`.claude/hooks/`) — скрипты на события:
`PreToolUse`, `PostToolUse`, `Stop`, `PreCompact`. Хуки могут **перехватывать**
ответы инструментов и модифицировать их до попадания в контекст — та же идея что
и прокси, но на уровне конфига.

### Пример

```
.claude/hooks/compact-test-output.sh:
  # PostToolUse — перехватывает ответ run_tests
  # Оставляет: pass/fail count + имена упавших тестов
  # Возвращает компактный JSON вместо сырого stdout
```

### Выигрыш

Эквивалентен Truncation и Field Filtering из прокси-каталога, но без отдельного
сервиса.

| Метод | Через прокси (Run 3) | Через хуки (Run 2+) |
|---|---|---|
| Truncation | ✅ | ✅ (PostToolUse) |
| Field Filtering | ✅ | ✅ (PostToolUse) |
| Structured Summary | ✅ | ✅ (требует парсера) |
| Pagination Enforcement | ✅ | ✅ (PreToolUse) |

### Сценарий

- Когда нет возможности развернуть отдельный прокси-сервис
- Быстрые трансформации без написания MCP-сервера
- **Не применять** для сложных трансформаций требующих состояния (Dedup, Aggregation)

## Комбинирование рычагов

Рычаги гигиены аддитивны. Типичный переход от раздутого AGENTS.md (Run 1) к
оптимизированному (Run 2):

```
Run 1 (раздутый):
  L0: Opus ($15/$75)
  L1: широкий scope → 20+ чтений
  L2: без ограничений → 5000 output токенов
  L3: без hints → 4 поисковых вызова
  L5: vague промпт → 6 ходов
  L6: CLAUDE.md 5000 chars → 1250 токенов/ход
  Итого: ~50K токенов, $3–5

Run 2 (гигиена):
  L0: Sonnet ($3/$15)
  L1: scope до 4 файлов → 5 чтений
  L2: «без преамбул» → 1200 output токенов
  L3: file hints → 0 поисковых вызовов
  L5: precise промпт → 2 хода
  L6: CLAUDE.md 400 chars → 100 токенов/ход
  Итого: ~12K токенов, $0.15–0.25

Совокупный выигрыш: ↓ 75% токенов, ↓ 93% стоимости.
```

## Потолок гигиены

Рычаги L0–L6 находятся **снаружи** цикла агента. L7–L10 (subagents, skills, lazy
tools, hooks) уже **частично контролируют цикл**, но всё ещё ограничены:

- Не могут сжимать ответы инструментов так же эффективно как прокси (хуки могут, но
  без состояния между вызовами)
- Не могут агрегировать пакетные запросы (требует удержания состояния)
- Не могут дедуплицировать с кешем между сессиями

Именно поэтому после Run 2 мы переходим к Run 3. Гигиена (L0–L10) даёт ↓ 75–90%
токенов. Оставшиеся потери — это раздутые MCP-ответы, которые эффективно чинит
только прокси.

## Как читать этот документ на воркшопе

- **Перед Run 2** — фасилитатор объясняет 2–3 базовых рычага (L0, L1, L2) и
  анонсирует advanced (L7–L10) как «следующий уровень».

  L7–L10 (subagents, skills, lazy tools, hooks) могли бы помочь.
- **После воркшопа** — полный каталог L0–L10 остаётся как reference для PlataCard.
