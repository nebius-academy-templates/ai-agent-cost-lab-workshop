# Каталог методов оптимизации хуков агента

Теоретический справочник по методам компактизации на уровне **хука агента** — скрипта,
который срабатывает **после** выполнения MCP-инструмента и переписывает его результат
до того, как тот попадёт в контекст модели. Каждый метод описан по единой схеме: логика,
трансформация, выигрыш, сценарий, частотность.

> **Связанные документы:** [charter.md](./charter.md) — общая архитектура воркшопа;
> [mcp-proxy-methods.md](./mcp-proxy-methods.md) — те же экономики, но на уровне прокси
> (важны различия — см. раздел «Чего хук не может»); [optimization-catalog.md](./optimization-catalog.md)
> — полный каталог всех рычагов.

---

## Где живёт хук

```
Прокси:   агент ──▶ [ ПРОКСИ ] ──▶ 5 MCP-серверов
                       ▲ man-in-the-middle: видит запрос И ответ, может всё

Хук:      агент ──▶ MCP-инструмент выполняется ──▶ [ ХУК ] ──▶ контекст модели
                                                     ▲ видит только РЕЗУЛЬТАТ, уже после вызова
```

Ключевое отличие от прокси: хук стоит **после** выполнения инструмента. upstream уже
сходил, уже вернул полный ответ, уже потратил время и токены на его генерацию. Хук может
только **переписать то, что вернулось**, до записи в контекст. Отсюда — и его сила
(простой, ничего не разворачивать, работает с любым агентом), и его потолок (не может
тронуть схемы инструментов, не может объединять вызовы — см. ниже).

---

## Модель стоимости

Экономика та же, что у прокси (хук компактит результат **до** записи в контекст и кеш):

- **Output (×5 стоимости input)** — компактный результат инструмента → модель пишет меньше.
- **Контекст переотправляется каждый ход** — результат в 200K символов, полученный на 3-м
  ходу, переотправляется на ходах 4, 5, 6… N. Хук режет это один раз, экономия — на всех ходах.
- **Cache write (×1.25) + cache read (×0.1)** — хук переписывает результат до того, как он
  попадёт в систему кеширования, поэтому в кеш ложится уже компактная версия.

```
Без хука: S × (1 cache-write + N × cache-read)
С хуком:  s × (1 cache-write + N × cache-read),  где s ≪ S

При S = 200K, s = 1K, N = 5:  ~350K → ~1.75K токен-единиц,  выигрыш ~200×
```

Разница с прокси — **только** в схемах инструментов (метод 6 у прокси): они отправляются
каждый ход, но хук их не видит. См. «Чего хук не может».

## API хука по агентам

Хук — это **отдельный процесс на каждый вызов** (без памяти между вызовами). На stdin
приходит JSON-событие; результат переписывается через stdout. Канал замены различается:

| Агент | Событие (регистрация) | Канал замены | Режим |
|---|---|---|---|
| **Claude Code** | `PostToolUse`, matcher `mcp__.*` (в `.claude/settings.json`) | `hookSpecificOutput.updatedToolOutput` | прозрачная перезапись |
| **Cursor** | `afterMCPExecution` (в `.cursor/hooks.json`) | печать события в stdout, `exit 0` | прозрачная перезапись |
| **Codex** | `PostToolUse`, matcher `mcp__.*` (в `.codex/hooks.json`) | `{ decision: "block", reason: "<компакт>" }` | **подмена через block** (прозрачного канала нет) |

Форма stdin (с вариациями): `{ tool_name: "mcp__<server>__<tool>", tool_input, tool_response \| tool_result \| response \| result, session_id }`.

**Важно про Codex:** у Codex в этой связке нет прозрачного «перепиши результат». Reference-хук
эмулирует подмену через `decision: "block"` с компактным `reason` — и только когда компактная
версия короче; иначе молчит и пропускает оригинал. Это рабочий, но менее чистый путь, чем у
Claude/Cursor.

**Принцип безопасности (все агенты):** хук переписывает результат **только если стало короче**,
иначе — молчит и оригинал остаётся. Любой сбой/невалидный вход → passthrough.

## 1. Filler Stripping (срезание болванки)

### Логика

Моки воркшопа подмешивают в длинные текстовые тела повторяющийся маркетингово-комплаенсный
«хвост». Он семантически пуст, но переотправляется каждый ход. Срезается одним regex.

### Трансформация

```
Вход (body, ~8K):
  "...реальный контракт...
   This endpoint is part of the enterprise integration suite. For SLA tiers,
   pagination semantics, ETag caching, rate-limit headers… [ещё 6K болванки]"

Выход (~2K):
  "...реальный контракт..."   // хвост от «This endpoint is part of…» удалён
```

Reference: `stripFiller(s)` — `s.replace(/\n*\bThis endpoint is part of the enterprise integration suite[\s\S]*$/g, "")`.

### Выигрыш

| Тело | После strip | Сокращение |
|---|---|---|
| confluence body с хвостом | ~70% хвоста | ↓ 60–75% |
| jira description с хвостом | ~50% хвоста | ↓ 40–60% |

### Сценарий

- Любой ответ с известной болванкой-хвостом (confluence/jira тела в этом воркшопе)
- В реальном проекте — баннеры, дисклеймеры, повторяющиеся преамбулы API
- **Не применять** к телам без известного шаблона болванки (можно срезать значимое)

### Частотность

~30 из 100 вызовов в этом воркшопе (моки везде добавляют хвост). В проде зависит от API.

## 2. Field Filtering / Mapping (фильтрация полей)

### Логика

Оставить только семантически значимые поля, отбросить навигацию/метаданные/аудит.
Рекурсивный strip известных мусорных ключей + per-server компакторы, оставляющие ровно
то, что нужно кодеру для фикса.

### Трансформация

```
Вход — jira issue (40+ полей, fields.* + comment + watchers + metadata):
  { key, fields: { summary, description, definitionOfDone, relatedResources,
                   labels, status: {name, iconUrl, ...}, comment: {...},
                   metadata: {...}, createdBy, lastReviewedBy, watchers: [...],
                   self, accountId, ... } }

Выход (8 полей):
  { key, summary, description, definitionOfDone, relatedResources, labels,
    status, comments: [{ body }] }
```

Reference: `STRIP_KEYS` (metadata, createdBy/At, lastReviewedBy/At, ownerTeam,
correlationId, auditTrailUrl, viewCount, self, iconUrl, accountId, html_url, node_id, …),
`STRIP_ARRAYS` (contributors, watchers) + `compactIssue/compactSentry/…`.

### Выигрыш

| Ответ | Полей всего | После | Сокращение |
|---|---|---|---|
| jira issue | 40+ | 8 | ↓ 75–85% |
| sentry event | 30+ | 6 + 5 фреймов | ↓ 80% |
| testrail run | 20+/кейс | 3/кейс | ↓ 75% |
| github statuses | 15+/статус | 3/статус | ↓ 80% |

### Сценарий

- Любой структурный JSON-ответ MCP с избыточными метаданными
- Per-server компактор там, где знаешь форму ответа (jira/confluence/sentry/testrail/github)
- Generic `stripBloat` (по `STRIP_KEYS`) — для незнакомых форм
- **Не применять**, когда агент явно просил конкретное «мусорное» поле

### Частотность

~25 из 100 вызовов. Почти все list/get-ответы несут навигационные/аудит-поля.

## 3. Truncation (усечение)

### Логика

Обрезать длинное текстовое тело до N символов с явным маркером `... [truncated by hook]`.
Самый дешёвый метод — не требует знания формата.

### Трансформация

```
Вход (body, 201K):  "...200K текста..."
Выход (~6K):        "...первые 6000 символов...\n\n... [truncated by hook]"
```

Reference: `truncate(s, 6000)` для общих тел, `5000` для confluence body.

### Выигрыш

| Размер тела | После | Сокращение |
|---|---|---|
| 100K+ | ~6K | ↓ 94%+ |
| 10–100K | ~6K | ↓ 40–94% |
| <6K | без изменений | 0% |

### Сценарий

- Длинные тела документов, логи, stdout — где хватает начала
- Fallback, когда JSON не распарсился (`compactToolResponse` → `truncate(stripFiller(t))`)
- **Не применять** там, где критична каждая строка (diff под apply_patch)

### Частотность

~35 из 100 вызовов. Самый частый — как fallback почти везде.

## 4. Structured Summary (структурное резюме)

### Логика

Распарсить ответ известного формата и вернуть **семантическое резюме** вместо сырого тела.
Самый «умный» метод — требует парсера под формат.

### Трансформация

```
Вход — testrail run (полные кейсы + стек + комментарии, ~40K):
  { runId, scenario, summary, results: [ {test_id, case_id, title, status,
      comment, stacktrace, ...80 кейсов } ] }

Выход (~1K):
  { runId, scenario, summary,
    results: [ { test_id, title, status } × N ] }   // только пасс/фейл + id
```

Reference: `compactTestrail` (results → `{test_id,title,status}`), `compactConfluence`
(body → `{title, excerpt}` или `truncate(body,5000)`), `compactSentry` (entries → первые 5
in-app фреймов стека).

### Выигрыш

| Формат | Размер | После | Сокращение |
|---|---|---|---|
| testrail run (80 кейсов) | 40K | ~1K | ↓ 97% |
| sentry event (полный стек) | 20K | ~1.5K | ↓ 92% |
| confluence страница | 200K | ~2K | ↓ 99% |

### Сценарий

- `testrail_*` → пасс/фейл + упавшие; `sentry_*` → culprit + верх стека; `confluence_*` → title + excerpt
- Любой ответ с известной схемой, где резюме достаточно для решения
- Требует парсер под каждый формат (дороже в реализации, чем truncation/field-filter)

### Частотность

~20 из 100 вызовов. Применим к ответам с известным форматом.

## 5. Item Slicing (пост-фактум пагинация)

### Логика

upstream вернул 30–100 элементов — хук оставляет первые N. Аналог пагинации прокси, но
**после** запроса: контекст ужимается, однако upstream уже потратился на полный ответ (см.
«Чего хук не может»).

### Трансформация

```
Вход — confluence search (30 результатов):
  { query, totalCount, results: [ ...30 по 1K... ] }     // ~30K

Выход (первые 5):
  { query, totalCount, returnedCount: 5,
    results: [ {id, title, excerpt, url} × 5 ] }          // ~3K
```

Reference: `p.results.slice(0, 5)` (confluence), `frames.slice(0, 5)` (sentry).

### Выигрыш

| N оставлено | Сокращение коллекции |
|---|---|
| 5 из 30 | ↓ ~83% |
| 5 из 100 | ↓ ~95% |

### Сценарий

- search/list-ответы, где агенту хватает первых 3–5
- **Не применять**, когда нужен полный набор (агрегация/подсчёт по всем элементам)
- Помнить: экономия — только на контексте, не на стоимости upstream-запроса

### Частотность

~25 из 100 вызовов (list/search).

## 6. Dedup / Cache (дедупликация — stateless-воркэраунд)

### Логика

Если тот же инструмент вызван с теми же аргументами — вернуть крошечный маркер
`{ _unchanged: true }` вместо тела. **Проблема:** хук — процесс без памяти между вызовами,
поэтому состояние **подделывается файлом** в `tmp`, ключ — `sha256(tool + input)`.

### Трансформация

```
Первый вызов  jira_get_issue(JIRA-0410): полный компактный ответ + запись ключа в файл
Повторный вызов (те же args):
  { _unchanged: true, _note: "identical to an earlier call this session — result omitted" }
  (~80 символов)
```

Reference: `dedupMarker(name, input, sessionId)` → файл `tmpdir()/<agent>-hook-dedup-<sessionId>.json`.

### Выигрыш

| Сценарий | Первый | Повтор | Экономия |
|---|---|---|---|
| повторный `get_issue` | компакт-ответ | ~80 chars | ↓ 95%+ на повторе |

### Сценарий

- Агент перечитывает тот же тикет/файл/тест в рамках сессии
- **Минусы относительно прокси** (у прокси кеш в памяти): racy при параллельных
  инструментах; файл остаётся в `tmp`; ключ только по точному совпадению input
- Без `session_id` — не работает (нечем разделить сессии)

### Частотность

~15 из 100 вызовов — повторные.

## 7. Diff-based Response (дифференциальный ответ)

### Логика

При повторном вызове polling-инструмента вернуть только **изменения** относительно прошлого
ответа. Та же стейтфул-проблема, что и dedup: предыдущий ответ надо где-то хранить (файл).

### Трансформация

```
Первый  get_test_result(RUN-001): { status: "running", passed: 0, ... }
Повтор:  { _diff: { status: "failed", passed: 96, failed: 24 }, _base: "prev" }
```

### Выигрыш

| Полный ответ | Diff | Сокращение |
|---|---|---|
| 5K (тесты) | ~0.3K | ↓ 94% |

### Сценарий

- Поллинг статуса (`get_test_result`, `*_status`)
- **Не применять** на первом вызове (нет базы) и при качественно другом ответе
- Требует хранения прошлого ответа (см. оговорки dedup)

### Частотность

~10 из 100 вызовов. Применим к polling-паттернам.

## Чего хук не может (асимметрия с прокси)

Главная причина существования этого документа: **не все методы прокси переносятся на хук.**

| Метод прокси | Хук? | Почему |
|---|---|---|
| **Schema Compaction** (описания инструментов) | ❌ нет | Схемы регистрируются MCP-сервером и шлются **каждый ход**. `PostToolUse`/`afterMCPExecution` видят только **результаты**, не реестр инструментов. Раздутые описания хуком не тронуть. |
| **Aggregation / Batching** | ❌ нет | Хук срабатывает **на один уже выполненный вызов**. Он не управляет диспетчеризацией и не может слить N вызовов в один. |
| **Pagination Enforcement** | ⚠️ частично | Хук режет уже вернувшуюся коллекцию (метод 5), но upstream **уже сходил и заплатил** за полный ответ. Экономия только на переотправке в контекст, не на upstream-стоимости/латентности. |
| **Dedup / Diff (стейт)** | ⚠️ воркэраунд | Память между вызовами подделывается файлом в `tmp` — racy, протекает, требует `session_id`. У прокси состояние в памяти, чисто. |
| Truncation / Field Filter / Summary / Filler | ✅ да | Работают по вернувшемуся результату — ровно то, что у хука есть. |

**Вывод:** если бюджет «съедает» раздутые **описания инструментов** (много инструментов,
многословные схемы) — это лечится **только прокси** (Schema Compaction). Если боль — в
размере **результатов** — хук решает её почти так же эффективно, как прокси, и без отдельного
сетевого сервиса.

## Комбинирование методов

Один результат `mcp__jira__get_issue` проходит несколько трансформаций подряд внутри хука
(reference: `dedupMarker(...) ?? compactToolResponse(...)`, где compact = field-filter +
summary + filler + truncation):

```
1. Dedup        → если повтор → {_unchanged:true} (выход, ↓ 100%)
2. Field Filter → fields.* (40 полей) → 8 значимых
3. Filler Strip → срезать «enterprise integration suite…» из description
4. Truncation   → оставшиеся длинные строки до 6000

Совокупно на одном вызове:
  Без хука: ~40K chars в контексте
  С хуком:  ~1.5K chars
  Выигрыш: ~25×
```

Порядок: dedup первым (самый дешёвый — может вообще снять работу), затем структурные
методы. Перезапись применяется, только если итог короче оригинала.

## Как читать этот документ на воркшопе

- **До Run 3** — фасилитатор объясняет, что хук переписывает результат **после** вызова, и
  чем он отличается от прокси (таблица «Чего хук не может»).
- **Во время Run 3 (Option B — хуки)** — участники заполняют scaffold (`hooks:setup` ставит
  passthrough, `hooks:solution` — эталон). Обязательны **Field Filtering** и **Truncation**;
  остальные — по выбору. Codex: помнить про `decision:"block"` вместо прозрачной перезаписи.
- **После воркшопа** — reference для построения реальных PostToolUse/afterMCPExecution хуков:
  быстрый способ ужать результаты MCP без отдельного прокси-сервиса.
