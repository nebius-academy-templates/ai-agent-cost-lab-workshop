# Каталог методов оптимизации MCP-прокси

Теоретический справочник по методам трансформации на уровне прокси. Каждый метод
описан по единой схеме: логика, выигрыш, сценарий, частотность.

> **Связанные документы:** [charter.md](./charter.md) — общая архитектура воркшопа;
> [optimization-catalog.md](./optimization-catalog.md) — полный каталог всех рычагов
> (включая гигиену и цикл агента).

---

## Модель стоимости

Прежде чем обсуждать методы — почему прокси вообще даёт эффект:

- **Output (×5 стоимости input)** — каждый символ ответа модели стоит в 5 раз дороже
  входного. Компактный ответ инструмента → модель пишет меньше → экономия на output.
- **Контекст переотправляется каждый ход** — ответ инструмента в 200K символов,
  полученный на 3-м ходу, будет заново отправлен на ходах 4, 5, 6… N.
- **Cache write (×1.25 input) + cache read (×0.1 input)** — даже если ответ попадает
  в кеш, его запись стоит дороже чем input, а чтение дёшево но не бесплатно. Прокси
  отсекает мусор до того, как он попадёт в систему кеширования.

**Формула эффекта для одного вызова инструмента с ответом размера S и N последующих ходов:**

```
Без прокси: S × (1 cache-write + N × cache-read)
С прокси:   s × (1 cache-write + N × cache-read),  где s ≪ S

При S = 200K, s = 1K, N = 5:
  Без прокси: 200K × (1.25 + 5×0.1) = 200K × 1.75 ≈ 350K токен-единиц
  С прокси:     1K × (1.25 + 5×0.1) =   1K × 1.75 ≈ 1.75K токен-единиц
  Выигрыш: ~200×
```

## 1. Truncation (усечение)

### Логика

Обрезать ответ до первых N символов / M элементов. Остаток доступен через явный
запрос артефакта. **Не тупое обрезание** — агент получает сигнал `_truncated: true`
и путь к полному телу.

### Трансформация

```
Вход (201K chars):
  { "name": "README.md", "content": "...200K текста..." }

Выход (967 chars):
  { "name": "README.md", "size": 201544, "preview": "...первые 800 символов...",
    "_truncated": true, "_artifact": "get_file_contents(path, artifact='full')" }
```

### Выигрыш

| Размер ответа | После truncation | Сокращение |
|---|---|---|
| 100K+ (README, логи) | ~1K | ↓ 99% |
| 10K–100K (списки, JSON) | ~2K | ↓ 80–98% |
| <1K (короткие ответы) | без изменений | 0% |

### Сценарий

- `get_file_contents`, `get_page`, `get_test_result` — любой вызов возвращающий
  большие текстовые тела
- Логи сборки, stdout/stderr тестов, содержимое документов
- **Не применять** к ответам где критична каждая строка (например, `git diff` для
  последующего `apply_patch`)

### Частотность

~35 из 100 вызовов. Агенты часто читают файлы и документы «на всякий случай» — в 9
из 10 случаев preview достаточно.

## 2. Field Filtering / Mapping (фильтрация полей)

### Логика

Оставить только семантически значимые поля, отбросить метаданные. В отличие от
truncation — не обрезает, а **прореживает** структуру, сохраняя её целостность.

### Трансформация

```
Вход (50+ полей):
  { "id": 952238700, "node_id": "R_kgDOOMICbA", "name": "mcp",
    "full_name": "awslabs/mcp", "private": false,
    "owner": { "login": "awslabs", "id": 3299148, "node_id": "...",
               "avatar_url": "...", "gravatar_id": "", "type": "Organization",
               "site_admin": false },
    "html_url": "...", "description": "Open source MCP Servers for AWS",
    "fork": false, "url": "...", "forks_url": "...",
    "keys_url": "...", "collaborators_url": "...",
    "teams_url": "...", "hooks_url": "...",
    ...ещё 30 полей...,
    "stargazers_count": 1420, "watchers_count": 1420,
    "language": "Python", "has_issues": true,
    "has_projects": true, "has_downloads": true,
    "has_wiki": true, "has_pages": false,
    "license": { "key": "apache-2.0", "name": "Apache License 2.0",
                 "spdx_id": "Apache-2.0", "url": "...", "node_id": "..." },
    "topics": ["aws", "mcp", "server"], ... }

Выход (6 полей):
  { "name": "mcp", "full_name": "awslabs/mcp",
    "description": "Open source MCP Servers for AWS",
    "stars": 1420, "language": "Python",
    "license": "Apache-2.0" }
```

### Выигрыш

Зависит от «коэффициента мусора» — доли неинформативных полей в ответе.

| Тип ответа | Полей всего | После фильтрации | Сокращение |
|---|---|---|---|
| GitHub repo (search) | 50+ | 6 | ↓ 85% |
| GitHub issue | 30+ | 5 | ↓ 80% |
| Jira ticket | 40+ | 7 | ↓ 75% |
| Slack message | 20+ | 4 | ↓ 70% |

### Сценарий

- `search_repositories`, `list_issues`, `list_pull_requests` — любые list/search
  ответы с избыточными метаданными
- Ответы API где 80% полей — `url`, `*_url`, `node_id`, `avatar_url` и другой
  навигационный мусор
- **Не применять** когда агент explicitly запросил конкретное поле

### Частотность

~25 из 100 вызовов. Большинство list/search ответов содержат навигационные поля,
нужные REST API но не агенту.

## 3. Structured Summary (структурное резюме)

### Логика

Распарсить ответ и вернуть **семантическое резюме** вместо сырого тела. Самая
«умная» трансформация — требует знания формата ответа.

### Трансформация

```
Вход (JUnit XML, 98K chars):
  <?xml version="1.0"?>
  <testsuites name="catalog-search" tests="120" failures="24">
    <testcase classname="catalog-search" name="should handle valid input..."
              time="0.045">
    </testcase>
    <testcase classname="catalog-search" name="should handle invalid input..."
              time="0.032">
      <failure message="Expected value to be within acceptable range">
        at Object.<anonymous> (/workspace/tests/catalog-search/scenario-1.spec.ts:42:10)
        ...ещё 20 строк стека...
      </failure>
    </testcase>
    ...ещё 118 testcase...
  </testsuites>

Выход (569 chars):
  { "runId": "RUN-catalog-search-1782234588296", "status": "failed",
    "total": 120, "passed": 96, "failed": 24, "durationMs": 45000,
    "failedTestIds": ["TC-catalog-search-0001", "TC-catalog-search-0006", ...],
    "_artifact": "full stdout/stderr/test details available on request" }
```

Другой пример — Markdown-документ:

```
Вход (README.md, 201K chars):
  # AWS MCP Servers
  ## Overview
  Suite of specialized MCP servers for AWS...
  ## Installation
  npm install @aws/mcp...
  ## Available Servers
  ### CDK Server
  ...
  ### Lambda Server
  ...
  [...ещё 190K...]

Выход (~1.5K chars):
  { "title": "AWS MCP Servers",
    "sections": ["Overview", "Installation", "Available Servers", "License"],
    "codeBlocks": 24, "totalLines": 4200,
    "summary": "Suite of specialized MCP servers for AWS services. Apache-2.0.
                Covers CDK, Lambda, ECS, Bedrock, DynamoDB, OpenSearch...",
    "_truncated": true }
```

### Выигрыш

| Формат ответа | Размер | После summary | Сокращение |
|---|---|---|---|
| JUnit XML (120 тестов) | 98K | 0.5K | ↓ 99.5% |
| Лог сборки (stdout) | 50K | 0.3K | ↓ 99.4% |
| Markdown-документ | 200K | 1.5K | ↓ 99.3% |
| JSON-конфиг | 5K | 0.8K | ↓ 84% |

### Сценарий

- `run_tests`, `get_test_result` — парсить JUnit/JSON-отчёт → пасс/фейл + список
  упавших
- `get_page`, `get_file_contents` — парсить Markdown → title + sections + code blocks
- Любой ответ с известной схемой где семантическое резюме достаточно для принятия
  решения агентом

### Частотность

~20 из 100 вызовов. Применим к ответам с известным форматом (тесты, документы,
конфиги). Требует написания парсера под каждый формат.

## 4. Aggregation / Batching (агрегация)

### Логика

Вместо N последовательных вызовов одного инструмента — объединить в один пакетный
вызов. Сокращает не размер ответа, а **количество вызовов** и накладные расходы на
каждый вызов (схема инструмента, JSON-RPC обвязка, model inference между вызовами).

### Трансформация

Агент хочет прочитать 5 файлов — делает 5 вызовов `get_file_contents`. Прокси
объединяет:

```
Без агрегации (5 вызовов):
  → get_file_contents("a.ts") → { content: "..." }
  → get_file_contents("b.ts") → { content: "..." }
  → get_file_contents("c.ts") → { content: "..." }
  → get_file_contents("d.ts") → { content: "..." }
  → get_file_contents("e.ts") → { content: "..." }
  Итого: 5 запросов к upstream, 5 ответов в контексте

С агрегацией (1 вызов):
  → get_files(["a.ts", "b.ts", "c.ts", "d.ts", "e.ts"])
  → { files: [{path: "a.ts", content: "..."}, ...] }
  Итого: 1 запрос к upstream, 1 ответ в контексте
```

### Выигрыш

| Сценарий | Без агрегации | С агрегацией | Сокращение |
|---|---|---|---|
| 5 файлов по 2K | 5 вызовов, 10K в контексте | 1 вызов, 10K в контексте | ↓ 80% вызовов, ↓ 0% данных |
| 10 файлов по 1K | 10 вызовов, 10K в контексте | 1 вызов, 10K в контексте | ↓ 90% вызовов |
| 3 search по 3K | 3 вызова, 9K в контексте | 1 вызов, 9K в контексте | ↓ 67% вызовов |

Выигрыш — в основном в сокращении количества ходов и model inference между вызовами.

### Сценарий

- Агент читает несколько файлов подряд (очень частый паттерн)
- Агент делает несколько search-запросов с разными query
- **Не применять** когда агент принимает решение о следующем вызове на основе
  предыдущего ответа

### Частотность

~20 из 100 вызовов (могут быть сгруппированы в 5–7 пакетных). Наиболее эффективен
для паттерна «прочитай все файлы в директории».

## 5. Dedup / Cache (дедупликация)

### Логика

Если агент вызывает тот же инструмент с теми же аргументами — не ходить в upstream,
вернуть `{ unchanged: true }`. Или вернуть закешированный ответ помеченный как
`_cached: true`.

### Трансформация

```
Первый вызов:
  → search_repositories("mcp server")
  ← { totalCount: 120528, items: [...30 repos...] }
  (ответ сохранён в кеш прокси с ключом sha256("search_repositories" + args))

Второй вызов (те же аргументы):
  → search_repositories("mcp server")
  ← { _unchanged: true, _cachedAt: "2026-06-23T21:00:00Z" }
  (0 токенов в upstream, ~50 chars в ответе)
```

### Выигрыш

| Сценарий | Первый вызов | Повторный вызов | Экономия |
|---|---|---|---|
| `search_repositories` | 3K ответ | 50 chars | ↓ 98% на повторе |
| `get_file_contents` | 200K ответ | 50 chars | ↓ 99.9% на повторе |
| `list_issues` | 5K ответ | 50 chars | ↓ 99% на повторе |

### Сценарий

- Агент перепроверяет результат теста (`get_test_result` с тем же runId)
- Агент заново открывает тот же файл после нескольких ходов
- Агент повторно ищет ту же документацию
- **Кеш живёт в рамках одной сессии** (in-memory, сбрасывается при рестарте агента)

### Частотность

~15 из 100 вызовов — повторные. Агенты склонны перепроверять и перечитывать.

## 6. Schema Compaction (сжатие описаний инструментов)

### Логика

Tool schemas (описания + input-схемы) отправляются **каждый ход**. Если upstream
MCP-сервер предоставляет многословные описания — они накапливаются в системном
промпте. Прокси перерегистрирует инструменты с компактными описаниями.

### Трансформация

```
Описание upstream (360 chars, каждый ход):
  "Performs a comprehensive full-text search across the entire Confluence
   knowledge base, indexing all spaces, pages, blog posts, comments, and
   file attachments. The search engine uses Apache Lucene under the hood
   with BM25 relevance scoring, supports complex boolean query syntax..."

Описание через прокси (80 chars, каждый ход):
  "Search Confluence documentation with full-text query."
```

### Выигрыш

| Количество инструментов | Раздутое описание (всего) | Компактное (всего) | Экономия в ход |
|---|---|---|---|
| 4 (воркшоп-моки) | ~4K chars | ~0.3K chars | ↓ 3.7K/ход |
| 26 (GitHub MCP) | ~1.2K chars | ~1.0K chars | ↓ 0.2K/ход |
| 10 средних | ~8K chars | ~0.8K chars | ↓ 7.2K/ход |

Умножается на количество ходов (10 ходов × 7.2K = 72K сэкономлено только на схемах).

### Сценарий

- Когда upstream MCP-серверы предоставляют избыточные описания (воркшоп-моки)
- Когда инструментов много (20+) и даже компактные описания в сумме значительны
- **Не даёт выигрыша** на уже хорошо спроектированных серверах (GitHub MCP — описания
  по 30–70 chars)

### Частотность

Применяется **один раз при старте прокси**, влияет на **каждый ход** агента.
Экономия растёт линейно с количеством ходов.

## 7. Pagination Enforcement (принудительная пагинация)

### Логика

Upstream API может возвращать 100+ элементов за раз. Прокси ограничивает `perPage`
до разумного значения (10–20) и добавляет курсор для следующей страницы.

### Трансформация

```
Без ограничения:
  → search_repositories("mcp")
  ← { totalCount: 120528, items: [...100 repos по 3K каждый...] }
  Ответ: ~300K chars

С ограничением (perPage=5):
  → search_repositories("mcp")
  ← { totalCount: 120528, returnedCount: 5, truncated: true,
      items: [...5 repos...], _nextPage: "search_repositories('mcp', page=2)" }
  Ответ: ~15K chars
```

### Выигрыш

| perPage | Размер ответа (search) | Сокращение |
|---|---|---|
| Без ограничения (30) | ~90K | — |
| 10 | ~30K | ↓ 67% |
| 5 | ~15K | ↓ 83% |
| 3 | ~9K | ↓ 90% |

### Сценарий

- `search_repositories`, `list_issues`, `list_pull_requests` — любые list-методы
  возвращающие потенциально большие коллекции
- Агент в 90% случаев использует только первые 3–5 результатов
- **Не применять** когда агент явно запросил большой perPage для анализа трендов

### Частотность

~25 из 100 вызовов. Большинство list/search вызовов могут быть ограничены.

## 8. Diff-based Response (дифференциальный ответ)

### Логика

При повторном вызове с изменившимися параметрами — вернуть не полный ответ, а
**разницу** с предыдущим.

### Трансформация

```
Первый вызов:
  → get_test_result("RUN-001")
  ← { status: "running", total: 120, passed: 0, failed: 0, ... }

Второй вызов (5 секунд спустя):
  → get_test_result("RUN-001")
  ← { _diff: { status: "failed", passed: 96, failed: 24 },
      _unchanged: ["total", "suite", "startedAt"], _base: "prev-response" }
  Вместо полного повторения — только изменения.
```

### Выигрыш

| Размер полного ответа | Размер diff | Сокращение |
|---|---|---|
| 5K (результаты тестов) | 0.3K | ↓ 94% |
| 3K (статус PR) | 0.1K | ↓ 97% |
| 10K (детали деплоя) | 0.5K | ↓ 95% |

### Сценарий

- Агент поллует статус операции (`get_test_result`, `get_pull_request_status`)
- Мониторинг: повторные вызовы `search_errors` с тем же query
- **Не применять** при первом вызове (нет базы для сравнения) и когда ответ
  качественно другой (не частичное обновление)

### Частотность

~10 из 100 вызовов. Применим к polling-паттернам. Требует хранения предыдущего
ответа в кеше прокси (см. Dedup).

## Комбинирование методов

Методы не исключают друг друга. Типичный вызов `search_repositories` может пройти
через 3–4 трансформации одновременно:

```
1. Schema Compaction → описание 360 → 80 chars (экономия каждый ход)
2. Pagination Enforcement → perPage 30 → 5 (↓ 83% данных)
3. Field Filtering → 50 полей → 6 полей (↓ 85% оставшегося)
4. Dedup → если повторный вызов → {_unchanged: true} (↓ 100%)

Совокупный эффект на одном вызове:
  Без прокси: ~300K chars в контексте
  С прокси:   ~500 chars в контексте
  Выигрыш: ~600×
```

- **До Run 3** — фасилитатор использует этот каталог для объяснения что такое
  прокси и какие методы доступны
- **Во время Run 3** — участники выбирают 2–3 метода для реализации в своём прокси
  (Truncation — обязательный, остальные на выбор)

  прокси в PlataCard
