# Методы оптимизации tool-layer

Единый справочник по всем методам оптимизации на уровне инструментов агента.
Для каждого метода явно указано, работает ли он в **Proxy-MCP** и/или **Local hook**.

> **Детали реализации:** [mcp-proxy-methods.md](./mcp-proxy-methods.md),
> [hooks-methods.md](./hooks-methods.md).

---

## Где стоит прокси, где стоит хук

```
                              каждый ход
         ┌──────────────────────────────────────────────┐
         │  системный промпт + схемы инструментов       │
         │           ▲                                  │
         │     ПРОКСИ перерегистрирует инструменты      │ ← Schema Compaction
         │     с компактными описаниями (1 раз)         │
         └──────────────────────────────────────────────┘
                              │
    Агент принимает решение и делает вызов инструмента
                              │
                              ▼
                      ┌──────────────┐
                      │  ПРОКСИ-MCP  │  ← man-in-the-middle
                      │              │     видит запрос И ответ
                      │  • меняет    │     может объединять вызовы (Aggregation)
                      │    запрос    │     ограничивает perPage до upstream
                      │  • кеш       │     (Pagination Enforcement)
                      │    in-memory │     кеш и dedup без файловых воркэраундов
                      └──────┬───────┘
                             │ модифицированный запрос
                             ▼
                    MCP-сервер (Jira, GitHub,
                    Confluence, Sentry…)
                             │ полный ответ
                             ▼
                      ┌──────────────┐
                      │  ПРОКСИ-MCP  │  ← трансформирует ответ:
                      │              │     Truncation, Field Filtering,
                      │              │     Structured Summary, Dedup, Diff
                      └──────┬───────┘
                             │ (без прокси — прямой путь сюда)
                             ▼
                      ┌──────────────┐
                      │     ХУК      │  ← PostToolUse / afterMCPExecution
                      │              │     видит только результат, уже после вызова
                      │              │     переписывает до записи в контекст
                      └──────┬───────┘
                             │
                             ▼
                    Контекст модели
```

**Ключевое различие:**
прокси стоит **между** агентом и MCP-сервером — видит запрос и ответ, может менять оба.
Хук стоит **после** выполнения — upstream уже сходил, хук может только переписать то, что вернулось.

---

## Модель стоимости

Ответ инструмента размером S, попавший в контекст на ходу 3, переотправляется на каждом
последующем ходу:

```
Без оптимизации: S × (1 cache-write + N × cache-read)
С оптимизацией:  s × (1 cache-write + N × cache-read),  где s ≪ S

Пример: S = 200K, s = 1K, N = 5  →  350K → 1.75K токен-единиц,  выигрыш ~200×
```

Оба подхода компактируют результат **до** записи в кеш — экономия одинаковая.
Разница — в том, **что именно** каждый может перехватить (см. таблицу ниже).

## Методы

### 1. Filler Stripping — срезание болванки

Срезает повторяющийся текстовый хвост (дисклеймер, SLA-секция, баннер) с помощью одного
regex, не разбирая структуру ответа.

```
Вход (~8K):  "...реальный контент...
              This endpoint is part of the enterprise integration suite.
              For SLA tiers, pagination semantics, ETag caching... [6K болванки]"

Выход (~2K): "...реальный контент..."
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ✅ |
| **Примечание** | Стандартный pipe после получения ответа | Самый дешёвый метод: один `s.replace(/паттерн/, "")` |

**Выигрыш:** ↓ 40–75%  
**Частота:** ~30 из 100 вызовов  
**Не применять:** к телам без известного шаблона болванки — риск срезать значимое

---

### 2. Field Filtering — фильтрация полей

Оставляет только семантически значимые поля, отбрасывает навигационные метаданные
(`*_url`, `node_id`, `avatar_url`, `createdBy`, `watchers`, `auditTrailUrl`…).

```
Вход — GitHub repo (50+ полей):
  { id, node_id, name, full_name, owner: {login, id, node_id, avatar_url, ...},
    html_url, forks_url, keys_url, ... ещё 35 полей ...,
    stargazers_count, language, license: { key, name, spdx_id, url, node_id } }

Выход (6 полей):
  { name, full_name, description, stars: 1420, language, license: "Apache-2.0" }
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ✅ |
| **Примечание** | Per-server компактор + generic strip по `STRIP_KEYS` | Те же компакторы; `dedupMarker ?? compactToolResponse` |

**Выигрыш:** ↓ 70–85%  
**Частота:** ~25 из 100 вызовов  
**Не применять:** когда агент явно запрашивал конкретное «мусорное» поле

---

### 3. Truncation — усечение

Обрезает длинное текстовое тело до N символов. Прокси сигнализирует `_truncated: true`
и путь к полному артефакту; хук добавляет маркер `… [truncated by hook]`.

```
Вход (README.md, 201K):  "...200K текста..."

Выход (~1–6K):
  Прокси: { name, size: 201544, preview: "...первые 800...", _truncated: true,
             _artifact: "get_file_contents(path, artifact='full')" }
  Хук:    "...первые 6000 символов...\n\n… [truncated by hook]"
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ✅ |
| **Примечание** | Отдаёт preview + путь к полному телу | Fallback по умолчанию для любого нераспознанного тела |

**Выигрыш:** ↓ 80–99%  
**Частота:** ~35 из 100 вызовов  
**Не применять:** там, где критична каждая строка (`git diff` под `apply_patch`)

---

### 4. Structured Summary — структурное резюме

Парсит ответ с известной схемой и возвращает семантическое резюме. Самый «умный»
метод — требует парсера под каждый формат, зато даёт максимальный выигрыш.

```
Вход — JUnit XML (98K, 120 тестов):
  <?xml ...> <testsuites tests="120" failures="24"> ... 118 testcase ... </testsuites>

Выход (~0.5K):
  { status: "failed", total: 120, passed: 96, failed: 24,
    failedTestIds: ["TC-0001", "TC-0006", ...], _artifact: "полный stdout по запросу" }
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ✅ |
| **Примечание** | Парсит на стороне прокси-сервиса | `compactTestrail`, `compactSentry`, `compactConfluence` |

**Выигрыш:** ↓ 84–99%  
**Частота:** ~20 из 100 вызовов  
**Не применять:** к ответам с незнакомой схемой; дешевле упасть на Truncation

---

### 5. Pagination / Item Slicing — ограничение коллекции

Ограничивает количество элементов в list/search-ответах. Прокси делает это **до**
запроса (меняет `perPage`) — upstream не генерирует лишнего. Хук делает это
**после** (Item Slicing) — срезает коллекцию уже полученного ответа.

```
Вход — search_repositories("mcp", perPage=100):  ~300K (100 репо × 3K)

Прокси (perPage=5, до upstream):
  { totalCount: 120528, returnedCount: 5, items: [5 репо], _nextPage: "..." }  ~15K

Хук (slice после upstream):
  { totalCount: 120528, returnedCount: 5, results: [5 репо] }                  ~15K
  ↑ upstream всё равно сходил за 100 репо
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ до upstream | ⚠️ после upstream |
| **Примечание** | Экономит и контекст, и стоимость upstream-запроса | Экономит только контекст; upstream уже заплатил за полный ответ |

**Выигрыш:** ↓ 67–90% контекста (у прокси — также upstream)  
**Частота:** ~25 из 100 вызовов  
**Не применять:** когда нужен полный набор для агрегации/подсчёта

### 6. Dedup / Cache — дедупликация повторных вызовов

Если агент вызывает тот же инструмент с теми же аргументами — вернуть `{_unchanged: true}`
вместо повторного полного тела.

```
1-й вызов: get_issue(JIRA-0410) → компактный ответ ~1.5K  (+ сохранить в кеш)
2-й вызов: get_issue(JIRA-0410) → { _unchanged: true }    ~50 chars
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ⚠️ |
| **Примечание** | In-memory кеш; быстро, чисто, без race conditions | Кеш через файл в `tmp` (ключ = `sha256(tool+input+sessionId)`); racy при параллельных вызовах, требует `session_id` |

**Выигрыш:** ↓ 98–99.9% на повторном вызове  
**Частота:** ~15 из 100 вызовов  
**Не применять:** когда ответ мог измениться между вызовами (состояние внешней системы)

---

### 7. Diff-based Response — дифференциальный ответ

При повторном вызове polling-инструмента возвращает только изменения относительно
предыдущего ответа.

```
1-й get_test_result(RUN-001): { status: "running", total: 120, passed: 0, ... }  5K
2-й get_test_result(RUN-001): { _diff: { status: "failed", passed: 96, failed: 24 },
                                 _unchanged: ["total", "suite"] }                 0.3K
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ⚠️ |
| **Примечание** | Предыдущий ответ хранится in-memory | Предыдущий ответ сохраняется в файл (те же ограничения, что у Dedup) |

**Выигрыш:** ↓ 94–97% на повторном вызове  
**Частота:** ~10 из 100 вызовов  
**Не применять:** на первом вызове (нет базы) и когда ответ качественно другой

---

### 8. Aggregation / Batching — объединение вызовов

Объединяет N последовательных вызовов одного инструмента в один пакетный запрос.
Сокращает количество ходов агента и накладные расходы на model inference между ними.

```
Без агрегации:
  → get_file_contents("a.ts") → ответ → get_file_contents("b.ts") → ... (5 ходов)

С агрегацией:
  → get_files(["a.ts","b.ts","c.ts","d.ts","e.ts"]) → [5 ответов]  (1 ход)
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ❌ |
| **Примечание** | Прокси перехватывает вызовы и управляет диспетчеризацией | Хук срабатывает на **уже выполненный** одиночный вызов; объединять нечего |

**Выигрыш:** ↓ 67–90% вызовов; меньше ходов → меньше model inference  
**Частота:** ~20 из 100 вызовов (→ 5–7 пакетных)  
**Не применять:** когда агент принимает решение о следующем вызове на основе предыдущего ответа

---

### 9. Schema Compaction — сжатие описаний инструментов

Перерегистрирует инструменты у агента с компактными описаниями. Схемы инструментов
отправляются **каждый ход** — экономия умножается на число ходов.

```
Описание upstream (360 chars × каждый ход):
  "Performs a comprehensive full-text search across the entire Confluence
   knowledge base, indexing all spaces, pages, blog posts, comments..."

Описание через прокси (80 chars × каждый ход):
  "Search Confluence documentation with full-text query."

10 инструментов × 10 ходов: ↓ 72K только на схемах
```

| | Прокси | Хук |
|---|---|---|
| **Поддержка** | ✅ | ❌ |
| **Примечание** | Перерегистрация при старте прокси; влияет на каждый ход | Хук видит **результаты** вызовов, не реестр инструментов; схемы недоступны |

**Выигрыш:** ↓ 0.2–7K/ход × N ходов  
**Частота:** применяется 1 раз, действует на каждый ход  
**Не применять:** на уже компактных серверах (GitHub MCP — описания 30–70 chars)
